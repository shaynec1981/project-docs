Your First Contribution

New here? This is the only doc you need before picking up your first issue. Keep it close for your first week.

Audience  New contributors Read time  ~5 min Questions  #programming on Discord
Section 01

Repo access

If you're reading this document, you most likely already have access to the GitHub repository — Form grants access to everyone who introduces themselves in the General channel after joining the server.

If you don't have access yet

Let Drykul or Form know in #programming and one of them will get you sorted.

Once you have access, clone the repository, open Godot 4.6, and import the project. Let the importer run on first open. Do not manually re-import assets that already exist in the project. If the project throws errors on a clean clone, report it in #programming before doing anything else.

Section 02

Five steps to your first merged PR

  1. 1
    Say hello in #programming

    This is where all CODE team activity happens — introductions, issue questions, architecture decisions. Let the team know you're ready to contribute.

  2. 2
    Clone the repo and verify it runs

    Clone the repo, open Godot 4.6, import the project, and confirm it runs without errors before touching anything. If it doesn't run cleanly on a fresh clone, post in #programming.

  3. 3
    Pick a "good first issue" from GitHub

    Browse open, unassigned issues and look for ones tagged good first issue. Self-assign the issue in GitHub when you're ready to start. See Section 03 for guidance on what to look for.

  4. 4
    Create a branch off code and do the work

    Always branch from code. Never work directly on code or Dev. Make your changes and write meaningful commit messages. Squash any WIP commits before opening a PR. The full branch workflow is in the contributor docs.

  5. 5
    Open a PR targeting code and wait for review

    Your PR must target the code branch. The lead programmer reviews every PR — do not merge your own, even if you have permission. If you haven't heard back in a reasonable time, ping in #programming.

You did it

Once your PR is merged, you're a contributor. Your guild rank reflects your skill level — don't sweat it if you're starting at the bottom. Everyone did.

Section 03

What issues should a first-timer pick?

A good first issue is self-contained — it doesn't require understanding the whole codebase to complete.

Issue type Good first issue? Why
Bug fix in a single script Yes Narrow scope, easy to test, clear pass/fail
Adding a new property or stat Yes Follows existing patterns — low risk
UI text or label change Yes Good way to learn the repo structure without breaking anything
Anything touching base classes No Base classes affect every extending script — ask in #programming first
Anything touching Autoloads No Changes cascade everywhere — not first-issue territory
Refactors or rewrites No Context-heavy — discuss in #programming before touching anything
When in doubt, ask

Post the issue link in #programming and say "Is this a reasonable first issue for me?" The team will let you know.

Section 04

Rules that matter most on day one

The full contributor docs go deep. These are the ones that will trip you up if you don't know them.

Do this
  • Branch from code, PR back to code
  • Self-assign an issue in GitHub before starting
  • Write meaningful commit messages
  • Use GameEvents for cross-script signals
  • Use GameData to read and write persistent data like stats and inventory
  • Extend base classes — never modify them
  • Post all questions in #programming
Never do this
  • Push directly to code or Dev
  • Start work on an already-assigned issue
  • Merge your own PR
  • Modify existing base classes
  • Re-import assets that already exist in the project
  • Leave WIP commits in a PR
  • Make architectural decisions alone — always discuss first
Section 05

Where to go when you're stuck

General code questions

Post in #programming on Discord. Describe what you're trying to do, what you've tried, and paste the relevant snippet. No question is too basic — this project exists partly to help people learn.

Issue-specific questions

Post in #programming, not in GitHub issue comments. Comments are for status updates. Discussions and decisions belong on Discord.

Architecture decisions

Always run these through #programming before touching anything. If it changes how systems connect or communicate, it's not a solo call.

Full contributor docs

Once you're settled, read the complete contributor guide. It covers the full GitHub workflow, coding standards, API documentation, and everything else in detail.

One rule above all others

When in doubt, ask in #programming before doing anything. A quick question saves everyone time.
