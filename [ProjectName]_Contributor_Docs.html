Section 01

About This Document

This document is the single source of truth for all code contributors on [PROJECT NAME]. It defines how work is picked up and delivered, how the project is structured, the architectural patterns all code must follow, and the standards your code is held to during review.

This is a living document. Sections marked ⚠ PENDING contain decisions still being finalized. Do not make assumptions about those areas — treat them as off-limits until this document is updated.

Proposing changes to this document

All proposed additions or corrections must go through the lead programmer (Drykul) via the #programming channel on Discord before any change is made. Do not submit PRs that modify this file without prior approval.

A note on PENDING sections

Section 02

Getting Started

Prerequisites

Before contributing, ensure you have the following installed and set up:

  • Godot 4.6 — use the standard build. Do not use the Mono/.NET version unless explicitly instructed.
  • GitHub Desktop — the recommended Git client for this project. Most contributors are assumed to be using it. If you prefer a different Git client or the command line, that is fine, but support in #programming will primarily be given for GitHub Desktop.
  • A GitHub account — required for issue assignments and pull requests. Make sure you have been granted repository access before attempting to clone.
  • A code editor — Godot's built-in script editor is sufficient for most work. VS Code with the Godot Tools extension is also commonly used and provides better autocompletion for GDScript.

Project Setup

  1. In GitHub Desktop, go to File → Clone Repository, find [PROJECT NAME], and clone it to a location on your machine.
  2. Open Godot 4.6 and import the project by clicking Import and pointing it at the project.godot file inside your cloned folder.
  3. Allow the Godot importer to run on first open. This is normal and may take a moment. Do not manually re-import assets that already exist in the project.
  4. Verify the project opens and runs without errors before picking up any work. If you see errors on a clean clone, report them in #programming before doing anything else.
#programming is your home base

All questions about issues, architecture decisions, file placement, or anything code-related must go through the #programming channel on Discord. Do not use GitHub issue comments as a substitute for a conversation that needs a decision from the lead programmer.

Section 03

GitHub Workflow

Full Branch Flow

Understanding the full picture helps you know where your work fits. The project uses a structured multi-stage branch flow:

feature branch → department branch → MERGE → dev

As a CODE team contributor, your work lives entirely in the first two stages. You create a feature branch off of code (the CODE team's department branch), do your work, and open a PR back into code. Everything beyond that — merging department branches into MERGE and integrating into dev — is handled by department leads at scheduled sync times.

About dev and the backup procedure

dev is the permanent unified build branch used for testing across all teams. There is no main branch. Before every integration of MERGE into dev, leads delete the previous backup and clone the current state of dev into a fresh standalone backup branch — then MERGE is integrated into dev. The backup is a safety snapshot, not a merge target. As a CODE contributor you will never interact with dev, MERGE, or backup directly.

Never target dev in your PR

All CODE team pull requests must target the code branch only. Pull requests targeting dev will be closed without review.

Picking Up an Issue

All work is tracked through GitHub Issues. There are two ways to receive work:

  • Direct assignment — Drykul assigns an issue to you and will notify you via #programming or a Discord DM. You do not need to monitor your GitHub notifications for this.
  • Self-selection — browse open issues and filter by the Code label. Any issue without an assignee is free to pick up. Before starting work, self-assign it in GitHub so others know it is taken.
Never work on an already-assigned issue

If an issue already has an assignee, do not begin work on it without coordinating in #programming first. Duplicate work will not be merged.

Branch Naming

All CODE team branches must follow this exact naming convention:

Feature-CODE-feature-name

The Feature prefix is always capitalized and always present. CODE is always uppercase and identifies this as a CODE team branch — other teams use their own designators (LEVDEV, ART, SOUND, etc.). The final segment must be short, all lowercase, hyphen-separated, and clearly describe the work being done.

Examples

Branch NameValidNotes
Feature-CODE-bauble-component✓Clear and correct
Feature-CODE-input-rebinding✓Descriptive, correct format
fix-bauble-bug✗Missing team code, wrong format
Feature-CODE-stuff✗Too vague — describe the actual work
Feature-ART-fish-sprites✗Wrong team code for a CODE branch

In GitHub Desktop: after cloning, go to Branch → New Branch, type the full branch name exactly as shown above, and make sure it is based on code. Drykul will delete your branch once it has been successfully merged.

Commit Messages

Commit messages are freeform — there is no enforced format. However, every message must be meaningful. It should describe what changed and why, not merely that something changed.

MessageValid
"Add BaubleComponent with biome access checking and GameEvents signal"✓
"Fix null reference in InputManager when no active context is set"✓
"stuff"✗
"fixed it"✗
"wip"✗ — see below

Squashing WIP commits before a PR

While working on a branch it is natural to make many small commits — "wip", "testing", "maybe this". Before opening a PR you need to squash those messy intermediate commits into one or a few clean, descriptive ones. This keeps the project history readable for everyone.

In GitHub Desktop:

  1. Click the History tab on your branch.
  2. Select the commits you want to combine — hold Shift to select a range, or Ctrl to select individual ones.
  3. Right-click the selection and choose Squash [X] commits…
  4. Write a single meaningful commit message describing the full change.
  5. Confirm. Your branch history is now clean.
Squash before opening the PR, not after

Once a PR is open and a review has started, rewriting history on the branch causes confusion. Squash your commits before you open the PR.

Pull Requests

When your branch is ready, open a PR in GitHub Desktop by clicking Branch → Create Pull Request. This opens GitHub in your browser. Before submitting, confirm the base branch is set to code, then notify Drykul via #programming or a DM on Discord.

Description

PR descriptions are intentionally lightweight. All that is required is a short, plain-language summary of what the PR does — a few sentences is enough. Write it at a level where someone unfamiliar with the specific issue can understand the gist of the change at a glance.

DescriptionValid
"Adds BaubleComponent. Checks player inventory for a specified bauble ID and emits a signal when a matching bauble is found. Wired to GameEvents."✓
"Fixed the input bug."✗ — too vague, describes nothing useful
(empty)✗ — always write something

If your PR closes a specific GitHub issue you are welcome to reference it (e.g. Closes #42), but this is not required.

Squash before you open

Make sure your WIP commits are squashed into meaningful commit messages before opening the PR. See Commit Messages for how to do this in GitHub Desktop.

Code Review & Merging

Drykul reviews and approves every pull request before it is merged. There are no exceptions. Do not merge your own PR under any circumstances, even if you have the repository permissions to do so. Drykul will also delete your feature branch once it has been merged.

During review you may receive change requests. Address all of them in new commits on the same branch — do not close and re-open a new PR. Once all changes are addressed, notify Drykul via #programming or DM. Do not use GitHub's re-request review button as a substitute for a message.

Review turnaround

This is a volunteer project. If your PR has not received a review within a reasonable time, send a message in #programming. Do not leave reminder comments on the PR itself.

The Co-location Rule

Co-location Rule

A .gd script lives in the same folder as its .tscn file if and only if that script is used exclusively by that one scene. If a script serves more than one scene, or has no associated scene at all, it belongs in scripts/.

FileLocationReason

BaseEntity.gdscripts/base/Extended by many scenes — belongs to none
GameEvents.gdscripts/autoloads/No associated scene — global singleton
AnimatedButton.gdscripts/ui/Used by multiple UI scenes

When in doubt about where a file belongs, ask in #programming before placing it.

Section 05

Architecture

Overview

All game code follows a three-layer architecture. Every script you write belongs to exactly one of these layers. Understanding which layer your work belongs to before writing any code is the most important habit to build as a contributor on this project.

Layer 1
Autoloads

Global singletons always accessible from any script. Strictly gatekept — never created without Drykul's approval.

Layer 2
Base Classes

Thin inherited scripts that give all entities a guaranteed interface. You may create new ones, but never modify existing ones without permission.

Layer 3
Components

Self-contained child scenes with documented extension points. The primary location for volunteer contributions.

The layers interact in one direction only: Layer 3 communicates upward via signals, either locally or through the GameEvents Autoload. No layer reaches downward to call methods directly on nodes it does not own.

Layer 1 — Autoloads

Autoloads live in scripts/autoloads/ and are globally accessible from any script by their name — no get_node() path required. They represent core systems that need to persist across all scenes and be reachable by unrelated parts of the project.

Do not create new Autoloads without approval

If you believe a system you are building qualifies as an Autoload, raise it in #programming before writing any code. Do not self-promote scripts to Autoload status. Creating an unauthorized Autoload will result in the PR being returned.

As a CODE contributor, your most common interaction with Layer 1 will be emitting and connecting to signals on GameEvents. See Section 07 — GameEvents for usage details.

Layer 2 — Base Classes

Base classes live in scripts/base/ and define the shared interface that groups of related entities must expose. The primary example used throughout this document is BaseEntity — note that this is a placeholder name used for illustration purposes. The actual class names used in this project will differ and will be documented in Section 08 as they are established.

Extending a base class gives your entity guaranteed lifecycle hooks and signals without having to define them from scratch. Every entity that shares behavior with other entities should extend the appropriate base class.

You can create new base classes

If you are building a group of entities that share behavior and no suitable base class exists yet, you are encouraged to create one. Place it in scripts/base/ and raise it in #programming so Drykul is aware of it.

Never modify an existing base class without permission

A change to an existing base class affects every entity that extends it — potentially the entire project. You must get explicit approval from the original author of the base class or from Drykul before making any changes. Raise it in #programming first.

The Player is a notable exception to the base class rule. Because the player character has a unique set of attributes and behaviors that no other entity shares, Player extends CharacterBody3D directly rather than a shared base class. This is intentional and is the only entity with this exemption.

GDScript# scenes/characters/Elias/Elias.gd
# Elias is an NPC entity — he extends a shared base class.
class_name Elias
extends BaseEntity

func _entity_ready() -> void:
    # Called by BaseEntity._ready(). Put entity init logic here.
    # Do not override _ready() directly in entities.
    pass

func _entity_die() -> void:
    # Called by BaseEntity.die(). Put cleanup logic here.
    pass

Layer 3 — Components

Components are self-contained child scenes attached to entities in the scene tree. This is where most volunteer contribution happens. A well-written component has no knowledge of its parent entity beyond what is passed to it through @export variables configured in the editor.

The example below is based on a planned BaubleComponent — baubles are inventory items that unlock gameplay features such as access to new biomes. The final implementation may differ, but the structural pattern shown here is what all components must follow regardless of what they do.

GDScript — planned example (BaubleComponent)# scenes/components/BaubleComponent/BaubleComponent.gd
class_name BaubleComponent
extends Node

# ── EXPORTS ───────────────────────────────────────────────────────────────
@export var required_bauble_id: String = ""
@export var unlocks_biome_id: String = ""

# ── SIGNALS ───────────────────────────────────────────────────────────────
signal biome_unlocked(biome_id: String)
signal access_denied()

# ── LIFECYCLE ─────────────────────────────────────────────────────────────
func _ready() -> void:
    pass

# ── PUBLIC INTERFACE ──────────────────────────────────────────────────────
## Checks whether the player's inventory contains the required bauble.
## Emits biome_unlocked if access is granted, access_denied if not.
## Does not modify the inventory — read only.
func check_access(inventory: InventoryContainer) -> void:
    if _inventory_has_bauble(inventory, required_bauble_id):
        biome_unlocked.emit(unlocks_biome_id)
        GameEvents.biome_access_granted.emit(unlocks_biome_id)
    else:
        access_denied.emit()

# ── PRIVATE ───────────────────────────────────────────────────────────────
func _inventory_has_bauble(inventory: InventoryContainer, id: String) -> bool:
    return false # implementation pending
Never call get_parent() to reach the owning entity

A component that calls get_parent() and then invokes methods on the result is tightly coupled to a specific parent type. It is no longer reusable or portable. Communicate outward through signals only.

Signals Primer

Signals are Godot's built-in event system. They allow one node to notify other nodes that something happened — without either node needing a direct reference to the other. This is the primary way components and systems communicate in this project.

Declaring a signal

Signals are declared at the top of a script, alongside your exports and variables. They can carry data as typed parameters.

GDScript — declaring signals# A signal with no data
signal cast_started()

# A signal that carries data
signal fish_caught(fish_name: String, weight: float)

Emitting a signal

Call .emit() on the signal when the event occurs. Any node that has connected to this signal will be notified automatically.

GDScript — emitting signalsfunc on_fish_reeled_in(fish_name: String, weight: float) -> void:
    fish_caught.emit(fish_name, weight)

Connecting to a signal

To listen for a signal, connect it to a function in _ready(). The function you connect to must accept the same parameters the signal carries.

GDScript — connecting to signalsfunc _ready() -> void:
    # Connect this node's own signal to a local function
    fish_caught.connect(_on_fish_caught)

    # Connect to another node's signal
    $FishingRod.cast_started.connect(_on_cast_started)

func _on_fish_caught(fish_name: String, weight: float) -> void:
    print("Caught a %s weighing %.1f kg!" % [fish_name, weight])

func _on_cast_started() -> void:
    pass

Local signals vs. GameEvents

Not every signal belongs on GameEvents. Use this rule to decide:

SituationUse
Only the parent entity cares about this eventLocal signal on the component
Two or more unrelated systems need to know about this eventGameEvents signal

For example, a FishingRodComponent emitting cast_started for the owning entity to animate — that is a local signal. A fish being caught that the UI, the achievement system, and the economy system all need to react to — that belongs on GameEvents.

GDScript — GameEvents usage# Emitting on GameEvents (from anywhere in the codebase)
GameEvents.fish_caught.emit(fish_name, weight)

# Connecting to a GameEvents signal in _ready()
func _ready() -> void:
    GameEvents.fish_caught.connect(_on_fish_caught)

func _on_fish_caught(fish_name: String, weight: float) -> void:
    pass

Coding Standards

Naming Conventions

All GDScript follows the official GDScript community naming standard without exception.

Identifier typeConventionExample
Variablessnake_casecast_distance, lure_weight
Functions (public)snake_casecheck_access(), get_stat()
Functions (private)_snake_case_calculate_bite_chance()
ConstantsSCREAMING_SNAKE_CASEMAX_CAST_DISTANCE, BASE_REEL_SPEED
Classes & nodesPascalCaseBaubleComponent, BaseEntity
Signalssnake_casefish_caught, biome_unlocked
EnumsPascalCaseBiomeType, LureCategory
Enum valuesSCREAMING_SNAKE_CASEBiomeType.DEEP_OCEAN

All function parameters and return types must be explicitly typed. Untyped code will be flagged during review. If you have a genuine technical reason to skip typing in a specific case, document it with an inline comment.

GDScript — naming examples# CORRECT
var lure_weight: float = 1.5
const MAX_CAST_DISTANCE: float = 50.0

func calculate_bite_chance(depth: float, bait_id: String) -> float:
    return 0.0

# INCORRECT
var LureWeight = 1.5            # PascalCase on a variable; untyped
var lw: float = 1.5             # Cryptic abbreviation
func CalcBite(d, b):            # PascalCase; untyped params; no return type
    return 0.0

File & Scene Naming

All .gd and .tscn files use PascalCase exactly matching the class_name or root node name they define. Folder names use lowercase with no spaces.

FilenameValid
BaubleComponent.tscn✓
BaubleComponent.gd✓
bauble_component.tscn✗
Baublecomponent.gd✗
Bauble Component.tscn✗ — spaces are never allowed in filenames or folders

Constants

Any value that appears more than once must be declared as a named constant. Magic numbers — raw numeric or string literals embedded directly in logic — make code harder to read, harder to tune, and impossible to search for project-wide.

GDScript — constants# CORRECT — single source of truth
const REEL_FORCE: float = 120.0

func on_reel_pressed() -> void:
    apply_tension(REEL_FORCE)

func on_reel_held() -> void:
    apply_tension(REEL_FORCE)

# INCORRECT — magic number used twice
func on_reel_pressed() -> void:
    apply_tension(120.0)

func on_reel_held() -> void:
    apply_tension(120.0)

Comments & Docstrings

Not every function needs a comment. A well-named function with clearly typed parameters is already self-documenting. Docstrings (using ##) are required only on functions where the why or how is not immediately clear from the code alone.

GDScript — comments# No docstring needed — the name says it all.
func get_current_lure_weight() -> float:
    return lure_weight

## Calculates the probability of a bite given depth and current bait.
## Returns 0.0 if the active biome does not support the bait type.
## Does NOT trigger a bite event — pass the result to attempt_bite().
func calculate_bite_chance(depth: float, bait_id: String) -> float:
    pass

Inline comments explain why, not what. If you feel the need to comment on what a line of code does, consider extracting it into a named function instead.

Typo Policy

If you encounter what appears to be a typo in existing code, do not change it without confirmation. What looks like a typo may be an intentional legacy name.

  1. Raise the suspected typo in #programming on Discord. Include the file path and line number.
  2. Wait for confirmation from Drykul.
  3. If confirmed, fix it in your current PR. Do not open a separate PR for a typo fix alone.

For code you are writing yourself, typos are your responsibility to catch before opening a PR.

Section 07

Autoload Reference

Autoloads are globally accessible from any script by their name alone. The following Autoloads are currently established in the project. All of them live in scripts/autoloads/.

GameEvents

GameEvents Global Singleton Also known as: EventBus

GameEvents is this project's EventBus — a central signal hub that allows unrelated systems to communicate without needing direct references to each other. It contains only signal declarations. No logic, no state, no functions except what signals require.

What belongs here: signals that cross system boundaries — things more than one unrelated system needs to know about. What does not belong here: signals that only a single parent cares about. Those stay local on the component or node that owns them.

Usage: See the Signals Primer in Section 05 for full examples of how to emit and connect to signals on GameEvents.

Ongoing refactoring — signals migration

The codebase is being refactored to centralize all cross-system signals into GameEvents. You may encounter signal declarations scattered across other scripts — this is legacy code being cleaned up, not a pattern to copy. All new cross-system signals must be declared in GameEvents.gd. If you are unsure whether your signal belongs there, ask in #programming.

InputManager

InputManager Global Singleton

Manages all player input through a context system. Only one input context is active at a time. Switching contexts is done by calling InputManager.set_mode() — never by modifying current_mode directly. Keybinds are loaded from user://controls.cfg automatically on startup.

Input modes

Available modes are defined in InputMode — a separate class intentionally, so you write InputMode.Movement rather than InputManager.InputMode.Movement. This is by design.

ModeWhen used
InputMode.MovementPlayer is in the world and can move
InputMode.MenuA menu is open
InputMode.InventoryThe inventory screen is open
InputMode.SettingsThe settings screen is open
InputMode.DisableAll input is suppressed (cutscenes, loading, etc.)

Switching modes

GDScript — switching input modes# Switch to inventory mode when a container is opened
InputManager.set_mode(InputMode.Inventory)

# Never do this — always use set_mode()
# InputManager.current_mode = InputMode.Inventory  ← WRONG

ItemManager

ItemManager Global Singleton

Manages all inventory interactions at runtime — picking up items, placing items, swapping items between slots, rotating and flipping held items, and updating container previews as the mouse moves. It is the single point of control for anything that happens to an item while a container is open.

ItemManager operates by querying the scene tree for nodes in the "held_item" and "inventory_container" groups each frame. It only processes input when GameState.containers_open is true.

Access pattern: ItemManager.add_item_to_player_inventory(item)

Public methods

Prints a warning if the inventory is full. This is the standard way to give the player an item from code. See Creating_New_Items_Guide.docx on the shared Google Drive for how to create ItemData resource files.

var apple: ItemData = preload("res://data/ui/resources/items/apple/apple.tres")

DebugOverlay

DebugOverlay Global Singleton Dev only — not in final build

A developer overlay that displays live runtime information — current FPS, player world position, and active input mode. It is toggled on and off by the toggle_debug_overlay input action, which emits InputManager.debug_toggle_pressed. The overlay is always hidden by default and only visible when explicitly toggled.

DebugOverlay will not be included in the final release build. It is available to all contributors during development as a useful diagnostic tool. You do not need to remove or modify it.

Editor Tuning Script

The Editor Tuning Script (in_editor_stats_adjustments.gd) is a @tool script attached to a node in the main level scene. It allows non-coders — designers, artists, and anyone else — to adjust tunable game values directly in the Godot Inspector without ever opening a script.

It works by mirroring all GameData values as @export variables. When a value is changed in the Inspector, a set() function pushes the new value directly into the GameData Autoload at runtime. The change is immediate and does not require reloading the scene.

For non-coders — how to tune values

Open the main level scene in Godot. Select the node with the Editor Tuning Script attached. In the Inspector panel on the right you will see categories for Player Movement, Camera, and Containers. Adjust values there and press Play to test. No script editing required.

For coders — adding new tunable values

When you add a new tunable value to GameData, you must also add a corresponding @export variable to the Editor Tuning Script so non-coders can adjust it. Follow this pattern:

GDScript — adding a new tunable value# Step 1: Add the value to GameData.gd
var cast_distance_max: float = 50.0

# Step 2: Mirror it in in_editor_stats_adjustments.gd
@export_category("Fishing")
@export var cast_distance_max: float = 50.0:
    set(value):
        cast_distance_max = value
        GameData.cast_distance_max = value

Inventory container configuration

The Editor Tuning Script also manages inventory container dimensions (rows and columns) for both player and shop inventories. These are exposed under the Containers category in the Inspector and emit an inventory_config_changed signal when changed so any listening UI can update itself.

The saved mode pattern

When switching to a temporary mode (inventory, settings) from which you need to return to wherever the player was before, store the current mode in saved_mode first, then restore it on exit. This is the established pattern used throughout the project.

GDScript — saved mode pattern# Opening a temporary mode — save current mode first
if InputManager.current_mode != InputMode.Inventory:
    InputManager.saved_mode = InputManager.current_mode
InputManager.set_mode(InputMode.Inventory)

# Closing that mode — restore saved mode
InputManager.set_mode(InputManager.saved_mode)

InputMode

InputMode is a separate class from InputManager — intentionally, so you write InputMode.Movement rather than InputManager.InputMode.Movement. Use InputMode.return_name(int) to get a human-readable string for any mode value, which is useful for debugging.

Enum valueIntWhen active
InputMode.Movement0Player is in the world and can move freely
InputMode.Menu1A menu is open
InputMode.Disable2All input is suppressed — cutscenes, loading, etc.
InputMode.Settings3The settings screen is open
InputMode.Inventory4The inventory screen is open

Input context classes

Each input mode has a corresponding context class that extends IInputContext. These are managed internally by InputManager — you do not instantiate them directly. The interface they implement is:

MethodCalled when
process() -> voidEvery frame while this context is active
on_enter() -> voidThis context becomes the active context
on_exit() -> voidThis context is replaced by another context

The one context with a public-facing method contributors may use is MovementInputContext:

Property / MethodTypeDescription

consume_interact() -> void—Clears interact_pressed after it has been handled, preventing double-processing.

PR Contribution Checklist

Verify every item before opening a pull request. PRs that fail basic checklist items will be returned without a full review.

Code Quality

  • All variables, functions, signals, and classes follow the naming conventions in Section 06
  • All function parameters and return types are explicitly typed
  • No magic numbers — any value used more than once is declared as a named constant
  • Complex or non-obvious functions have ## docstrings
  • No debug code left in — no stray print(), breakpoints, or commented-out blocks

Architecture

  • No new Autoloads created without prior explicit approval from Drykul
  • No component calls get_parent() to reach and invoke methods on its owning entity
  • Cross-system signals are declared in GameEvents.gd, not scattered across scripts
  • Input mode changes use InputManager.set_mode() — not direct assignment to current_mode
  • Any new base class is placed in scripts/base/ and flagged to Drykul in #programming
  • No existing base class was modified without permission from its original author or Drykul
  • All scripts are in the correct location per the Co-location Rule in Section 04

Files & Git

  • Branch name follows the Feature-CODE-description convention
  • Branch was created from code and the PR targets code — not dev
  • WIP commits have been squashed into meaningful commit messages before the PR was opened
  • No files were added to assets/art/item_shape_previews/ — that folder is restricted to the inventory extension only
  • All .tscn and .gd files use PascalCase filenames
  • No folder or file names contain spaces

Typos

  • No new typos introduced in variable names, function names, signals, or comments
  • Any suspected typos in existing code were confirmed via #programming before being changed

  • Branch naming: always Feature-CODE-your-description. No exceptions. (full rule)

Project Structure

  • Co-location rule: if a .gd script belongs to exactly one scene, it lives in the same folder as that .tscn. If it serves more than one scene or has no scene, it belongs in scripts/. (full rule)
  • File naming: PascalCase for all .gd and .tscn files. Lowercase for all folder names. No spaces anywhere. (full rule)
  • Restricted folder: assets/art/item_shape_previews/ is for the inventory extension only — do not add files there. (full rule)

Architecture

  • Three layers: Autoloads (global) → Base Classes (inherited) → Components (child scenes). Know which layer your work belongs to before writing. (full rule)
  • New Autoloads: never create one without Drykul's explicit approval. (full rule)
  • New base classes: allowed — place in scripts/base/ and tell Drykul. Modifying existing base classes requires permission from the original author or Drykul. (full rule)
  • Components: never call get_parent() to reach the owning entity. Communicate outward through signals only. (full rule)
  • Signals: if only one parent cares → local signal on the component. If two or more unrelated systems care → it belongs on GameEvents. (full rule)
  • Input mode switching: always use InputManager.set_mode(InputMode.X). Never assign current_mode directly. (full rule)

Coding Standards

  • Naming: snake_case for variables, functions, signals. PascalCase for classes, nodes, enums. SCREAMING_SNAKE_CASE for constants and enum values. All functions must have typed parameters and return types. (full rule)
  • Constants: any value used more than once must be a named constant. No magic numbers. (full rule)
  • Docstrings: required only on complex or non-obvious functions. Use ## directly above the function signature. (full rule)
  • Typos: never change a suspected typo without confirming with Drykul in #programming first. If confirmed, fix it in the same PR. (full rule)

  • GameEvents — the signal bus. Emit and connect cross-system signals here. (full reference)
  • GameState — live runtime state. Player body reference, location, money, UI flags. (full reference)
  • GameData — tunable values. Speeds, camera, and other game-feel numbers. (full reference)
